## 레볼루션 0: 환경 설정

### 이해도: 높음

**잘된 점:**
- 설치 명령어가 OS별로 표로 정리되어 한눈에 파악 가능
- 트러블슈팅 표가 실용적 — Mac 권한 오류, 명령어 인식 안됨 등 실제로 부딪힐 문제를 미리 안내
- "Explanatory 모드 권장"이 학습용이라고 명시되어 선택 근거가 명확
- Phase A/B 분리로 "설명 → 실습 → 확인" 흐름이 자연스러움
- PPTX 프로젝트 적용 예시가 있어서 "이걸 배우면 이런 걸 만들 수 있다"는 동기부여 존재

**막힌 지점:**
- `curl -fsSL https://claude.ai/install.sh | bash` — 비개발자에게 이 명령어 자체가 위협적일 수 있음. "이 명령어를 그대로 복사-붙여넣기 하세요"라는 안내가 있으면 좋겠음
- "터미널을 열고"라는 지시에서 터미널을 어떻게 여는지 설명 없음. Mac에서 Spotlight로 "터미널" 검색, Windows에서 PowerShell 여는 법 등 1~2줄 추가 필요

**개선 제안:**
- [P2] 터미널 여는 법 간단 안내 추가 (Mac: Cmd+Space → "터미널", Windows: 시작 메뉴 → "PowerShell")
- [P3] 명령어 복사-붙여넣기 팁 명시 ("아래 명령어를 그대로 복사하여 터미널에 붙여넣기 하세요")

## 레볼루션 1: 체험 — 먼저 느껴보기

- PPTX 예시("5장짜리 팀 소개 슬라이드 만들어줘")가 구체적이고 비개발자에게 매력적

**막힌 지점:**

**개선 제안:**
- [P2] EXECUTE 섹션에서 "AskUserQuestion을 써서"를 자연어 표현으로 변경 ("Claude에게 '좀 더 자세히 물어봐줘' 또는 '나한테 되물어봐'라고 요청해보세요")
- [P3] 데모 1을 자기주도 학습자도 따라할 수 있도록 구체적 실행 예시 보강

## 레볼루션 2: 왜 터미널인가?

### 이해도: 높음

**잘된 점:**
- GUI vs CLI 비교 다이어그램이 직관적
- "코딩 = 개발이 아닌 범용 지식 작업 도구"라는 메시지가 비개발자 심리적 장벽을 효과적으로 낮춤
- PPTX 수작업 vs 자동화 비교 표가 매우 설득력 있음 (2시간 vs 5분, 열 번째도 동일)
- 퀴즈 2개가 핵심 개념을 잘 점검 (CLI 필요성 + 비개발자 관련성)
- reference의 EXECUTE 섹션에 "지금 쓰는 업무 도구의 한계를 CLI로 해결할 방법" 질문이 실질적

**막힌 지점 — STOP 프로토콜 위반 (P1 구조적 이슈):**

`prompts/why-cli.md`의 Phase A/B 구조가 다른 레볼루션과 일관되지 않음:

| 레볼루션 | Phase A | STOP | Phase B |
|---------|---------|------|---------|
| Rev 0 | 설명 + 실습 안내 | STOP | 확인 Quiz |
| Rev 1 | 데모 + 실습 안내 | STOP | 반영 Quiz |
| **Rev 2** | **설명 + Quiz 1** | **STOP** | **Quiz 2 + 정리** |
| Rev 3 | 설명 + 실습 안내 | STOP | Quiz |
| Rev 4 | 설명 + 실습 안내 | STOP | Quiz 3연속 |
| Rev 5 | 설명 + 실습 안내 | STOP | 검증 Quiz |
| Rev 6 | 설명 + 실습 안내 | STOP | 반영 Quiz |

**구체적 문제점:**

1. **Phase A에 Quiz가 포함됨**: 다른 모든 레볼루션에서 Phase A는 "설명 + 실습 안내"이고, Quiz/확인은 Phase B에서 수행. Rev 2만 Phase A 안에 Quiz 1이 있음 (line 25-29). 이는 STOP 프로토콜의 핵심 의도("설명+실습을 먼저, 확인은 나중에")를 위반.

2. **Phase A에 EXECUTE(실습) 섹션 부재**: prompt 파일의 Phase A에 실습 안내가 없음. reference 파일에는 EXECUTE 섹션이 있으나, prompt가 이를 참조하지 않음. 다른 레볼루션은 "참조 파일의 EXECUTE 섹션을 읽고 실습을 안내하세요"라는 지시가 있지만, Rev 2 prompt에는 이 지시가 누락됨.

3. **Phase A 헤더 명칭도 상이**: `Phase A (설명 + Quiz 1)`로 되어 있어, 다른 레볼루션의 `Phase A (설명 + 실습 안내)` 패턴과 다름.

4. **비개발자 관점 영향**: Phase A에서 설명 직후 바로 Quiz가 나오면, "아직 실습도 안 했는데 퀴즈?"라는 당혹감 발생 가능. 다른 레볼루션에서는 설명 → 실습 → STOP → Quiz 순서라서 실습을 해본 뒤에 이해도를 점검하는 자연스러운 흐름.

**수정 제안 (P1):**

Phase A를 다른 레볼루션과 동일한 패턴으로 재구성:
```
Phase A (설명 + 실습 안내):
  - 핵심 설명 (GUI vs CLI, 비개발자와 코딩)
  - 참조 파일의 EXECUTE 섹션 안내
  - "위 내용을 직접 실행해보세요" 마무리 멘트
  - STOP

Phase B (퀴즈 + 정리):
  - Quiz 1: CLI 필요성
  - Quiz 2: 비개발자 관련성
  - 레볼루션 3 이동 안내
```

**개선 제안:**
- [P1] Phase A에서 Quiz 1을 제거하고 Phase B로 이동. Phase A에 EXECUTE 참조 안내 추가. 다른 레볼루션과 동일한 "설명 → 실습 → STOP → Quiz" 패턴 적용
- [P1] Phase A 헤더를 `Phase A (설명 + 실습 안내)`로 변경하여 일관성 확보

## 레볼루션 3: Claude Code 7대 핵심 기능

### 이해도: 보통~높음 (기능별 차이 있음)

**기능 1 (CLAUDE.md) — 이해도: 높음**
- "팀 위키" 비유가 비개발자에게 즉시 이해 가능
- `/init`과 `/memory` 명령어가 실행 가능하고 결과가 바로 보임
- PPTX 스타일 가이드 예시가 매우 구체적

**기능 2 (Skill) — 이해도: 높음**
- "요리 레시피" 비유가 효과적
- Progressive Loading 개념이 "필요할 때만 로드"로 잘 설명됨
- 로컬 vs 전역 스킬의 "사내 레시피 vs 나만의 레시피 북" 비유 명확
- PPTX theme-setter 예시가 레볼루션 5와 자연스럽게 연결

**기능 3 (MCP) — 이해도: 보통**
- "플러그처럼 꽂으면 바로 사용" 비유 효과적
- 실용적 예시 표(Slack, Calendar, Notion 등)가 비개발자에게 공감 유도
- 그러나 `.mcp.json` 코드 블록이 비개발자에게 다소 위협적 — "Claude에게 'MCP 서버를 추가해줘'라고 요청하면 자동으로 처리됩니다"라는 안내가 있어서 어느 정도 해소
- Node.js 안내가 있는 것은 좋으나, 비개발자에게 Node.js 설치 자체가 큰 허들일 수 있음

**기능 4 (Subagent) — 이해도: 높음**
- "상사-부하 관계" 비유가 비개발자에게 즉시 이해 가능
- 컨텍스트 절약 개념이 잘 전달됨
- PPTX 3분할(테마/내용/렌더링) 예시가 일관성 유지

**기능 5 (Agent Teams) — 이해도: 보통**
- Subagent와의 차이("상사-부하" vs "프로젝트 팀 + 칸반 보드")가 명확
- 그러나 활성화 방법(`settings.json` 수정, tmux 설치)이 비개발자에게 어려울 수 있음
- "실험적 기능"이라는 점을 사전에 알려주는 것은 적절
- tmux 미지원 환경 대안 표가 잘 정리됨

**기능 6 (Hook) — 이해도: 보통**
- "자동 체크리스트" 비유 효과적
- "확률적 vs 결정적" 구분이 새로운 관점 제공
- 그러나 JSON 설정 코드가 비개발자에게 부담 — 보안 경고까지 있어서 "이거 잘못하면 위험한 거 아닌가" 하는 불안 유발 가능
- 보안 주의사항은 필요하지만, 비개발자에게는 "Claude에게 '이런 Hook 추가해줘'라고 요청하면 안전하게 설정됩니다"가 선행되면 좋겠음

**기능 7 (Plugin) — 이해도: 높음**
- "한 줄로 팀 전체가 동일한 환경"이라는 가치 제안이 명확
- 설치/업데이트 명령어가 간단하여 비개발자도 접근 가능
- "이 프로젝트 자체가 Plugin" — 메타적 학습 효과

**전체 정리 다이어그램:**
- 7가지 기능의 관계도가 잘 구성됨
- "모르면 Claude에게 물어보면 됩니다" — 학습 부담 완화 매우 효과적
- 레볼루션 3 → 4 전환 안내 ("고급 기능을 먼저 체험, 기초를 나중에")가 자연스러움

**막힌 지점:**
- 기능 7개를 순서대로 하면 분량이 많아서 피로도 높을 수 있음. 단, 각 기능이 Phase A/B로 잘 분리되어 있어 중간 이탈 후 재개가 가능
- 기능 3(MCP), 5(Agent Teams), 6(Hook)에서 JSON 코드 블록이 반복 등장하며 비개발자에게 부담

**개선 제안:**
- [P2] 기능 3/5/6에서 JSON 코드 앞에 "아래 코드는 참고용이며, Claude에게 말로 요청하면 자동 처리됩니다" 안내를 일관되게 추가 (기능 3에는 이미 있으나, 5와 6에는 부족)
- [P3] 레볼루션 3 시작 시 "7개 기능 중 관심 있는 것부터 해도 됩니다" 안내 고려 (이미 "특정 기능 번호를 지정하면 해당 기능으로 이동" 있음)

## 레볼루션 6: 리서치와 학습의 결합

### 이해도: 보통

**잘된 점:**
- 전체 커리큘럼을 종합하는 위치 배치가 자연스러움
- domain-research 파이프라인(Step 0~Final)이 시각적으로 잘 표현됨
- 자기주도 학습 워크플로우(관심 주제 → 리서치 → 자동화 → 공유)가 체계적
- PPTX 파이프라인 예시("리서치 → PPTX 자동 변환")가 0~5의 모든 기능을 통합
- 학습 주제 예시 표가 다양한 방향을 제시
- 졸업 정리 체크리스트가 성취감 제공
- "학습은 끝이 아니라 시작입니다" — 마무리 메시지 적절

**막힌 지점:**
- `/research` 명령어가 실제로 domain-research 스킬을 호출하는 것인지, 일반 대화인지 비개발자 입장에서 혼동 가능. 실제 스킬명은 `domain-research`인데 `/research`로 표기되어 있음
- domain-research 파이프라인의 각 Step이 구체적으로 사용자에게 어떤 행동을 요구하는지 불분명. Step 0의 "대화형 의도 분석"이 사용자가 무엇을 하면 되는 건지 감이 잘 안 올 수 있음
- 반영 질문이 4개 선택지 모두 "맞다"이면 사실상 의미 있는 자기평가가 아닐 수 있음

**개선 제안:**

### PPTX 예시 연결성 평가

PPTX 프로젝트 예시가 레볼루션 0~6을 관통하며 일관된 스토리를 형성함:
- Rev 0: 프로젝트 폴더 생성 (기반)
- Rev 1: "5장짜리 슬라이드 만들어줘" (체험)
- Rev 2: 수작업 vs 자동화 비교 (동기부여)
- Rev 3: 각 기능별 PPTX 적용 (기능 학습)

    Subagent = 3분할 처리, Agent Teams = 3담당 협업, Hook = 품질 체크, Plugin = 팀 배포
- Rev 4: Git으로 PPTX 버전 관리
- Rev 5: pptx-maker 스킬 직접 제작
- Rev 6: 리서치 → PPTX 자동 변환 파이프라인

이 연결성은 우수하며, 비개발자가 "이걸 배우면 이런 걸 만들 수 있다"는 목표를 지속적으로 인식할 수 있게 해줌.

### 총평

전체적으로 비개발자를 위한 커리큘럼으로서 높은 완성도를 보임. Phase A/B 구조, 비유 활용, PPTX 관통 예시, "Claude에게 물어보면 됩니다" 철학이 일관되게 적용됨.
